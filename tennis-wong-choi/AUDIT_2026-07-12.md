## 2. 賽事評估：全部公式

### 現行邏輯 ✅

**骨幹 = Elo**（`features/elo.py:4-5`）：`P = 1/(1+10^((elo_b−elo_a)/400))`
- K-factor：`K = 120/(n+5)^0.4`（`calibration.py:21-23`；新人 K≈63，200 場 K≈13）
- 起始 1500；surface Elo 第一次見新場地時由 overall Elo 種子（`elo_builder.py:74-75`）

**合成器**（`probability_model.py:268-309`）— 唔係加權平均，係 logit 疊加：
```
base_logit = 0.65×logit(P_surface_elo) + 0.35×logit(P_overall_elo)   ← 真骨幹
total_nudge = Σ gain_i × (P_i − 0.5)，cap ±1.20 logit
P_final = clamp( sigmoid(base_logit + total_nudge), 0.02, 0.98 )
```

**Nudge gains（真正嘅權重，`probability_model.py:40-51`）**：serve_return 1.10｜recent_form 0.55｜opp_rank_bucket 0.55｜H2H 0.45（要 ≥3 場先生效）｜tournament_level 0.35｜pressure 0.30｜fatigue 0.30（每日休息差 ×0.015，band ±0.10）｜round_perf 0.20｜big_match 0.20｜**injury 0.0（死）**。缺數據嘅因素標 inactive，貢獻 0（唔會注入假 0.5 訊號 — 呢part處理正確）。

**小樣本收縮**：`shrink_rate = raw×min(n/20,1) + prior×(1−min(n/20,1))`（`common.py:46-48`），prior = 球員自己整體勝率（round_perf 例外：寫死 0.5）。

**定價鏈**（`pricing.py`）：
- 去水：`no_vig_a = implied_a/(implied_a+implied_b)`（proportional）
- **Edge = 模型機率 − 去水後市場機率**（`edge.py:5`）— 即係要贏「公平價」先算有 edge
- Segment risk：ITF/Challenger/未知級別 edge×(1−0.12)、250 級 ×(1−0.05)，同時將模型機率推向市場（`segment_risk.py`）
- 注碼：**tenth-Kelly**（`f*×0.10×100u`），下限 1u、上限 5u、0.5u 進位（`staking.py`、`config.py:106-110`）
- 信心分 = 資料質素分 + min(edge,0.10)×100，rank_seed −8（`bet_filter.py:102-116`）

**Props**（新引擎，07-04 上線）：
- Aces（`ace_model.py`）：近 15 場，`serve_est = 0.70×整體 + 0.30×場地`，對手接發 25% 混入；P(over) 用 **27,299/57,971 場實證校準曲線**（拒絕咗 Poisson/Normal）；line 超過預測平均 1.25× 直接拒絕（防假 edge）；兩邊精確去水；`edge≥0.04 且 EV>0` 先算 value
- Games（`games_model.py`）：用「賽事激烈度」closeness=1−|2p−1| 查表（19,406 場擬合），1.45× cap
- **Temper 誠實化**（`props/calibration.py`）：模型機率先向 0.5 收縮（未夠 120 條結算 = 15% 力度；scorecard 話模型贏 = 5%；輸 = 35%）先計 EV

**10 個 agents**（`agents/`，唔係 12 個 — `runner.py:15-26` 只掛 10 個）：全部只出「標籤＋警告＋否決」，**唔會改機率或注碼**。有實權嘅否決：data_quality invalid、injury D/E（死）、市場價低過最低可接受、final_decision 匯總。

### 已發現問題

1. ✅ **`WEIGHTS` dict 係純display**（`probability_model.py:9-26` 自己講明）— 報告會顯示「surface 0.22 / serve_return 0.20…」但實際影響力係 NUDGE_GAINS。睇報告嘅人會誤解權重。
2. ✅ **serve_return 係 placeholder 但 gain 最大（1.10）**：來源係 tournament-level 聚合 hold/break rate，唔係真正逐場發球模型（`feature_builder.py:129-132` 自己標明 Stage 3 placeholder）。
3. ✅ **Surface Elo fallback 造成重覆計算**：球員冇該場地紀錄時 surface Elo = overall Elo（`feature_builder.py:121`），兩個「獨立」骨幹變成同一個訊號食晒 100% 權重。
4. ✅ **信心分同資料質素分幾乎同義**（edge 貢獻 cap 到 +10，code 註解自認 edge 冇預測力）— 信心分講嘅係「數據幾齊」，唔係「幾大機會贏」。
5. ✅ 校準驗證用機率空間 0.65/0.35，live 用 logit 空間 — 驗證嘅嘢≠上線嘅嘢（數值差細但唔嚴謹）。
6. ✅ NUDGE_GAINS、segment 折扣、fatigue 0.015 全部係冇引用回測嘅 magic numbers（Elo K 同硬閘門就有回測依據）。
7. ✅ Kelly 1u 下限喺風險乘數之後先套用 → 細 edge 時所有減注乘數被下限抹平。

### 建議改動
- 報告移除／改正 WEIGHTS 顯示，直接顯示 nudge 貢獻。
- 清走 injury 死碼或者接真數據源（二揀一，唔好留殭屍）。
- Surface Elo fallback 時應該降低 surface 分量而唔係照抄 overall。

## 3. 揀注同組合邏輯

### 現行邏輯 ✅

**單注閘門鏈**（`bet_filter.py`，全部要過）：
質素分 ≥65 → provenance valid → 有賠率 → 賠率 ≥ 最低可接受（=1/(P−0.035)）→（賠率≥5.0 時質素 ≥80）→ 有齊 surface+overall Elo → edge 分級（<0.02 NO_BET｜<0.035 WATCHLIST｜<0.05 小注｜<0.08 標準｜≥0.08 大注）→ Kelly>0 → **edge≥0.20 反殺**（假 edge 位）→ **賠率≥5.0 反殺** → 信心 ≥65。

**組合**（`combos.py`）：最多 3 腳；同場對立方向禁止；相關性 haircut 平均 pairwise、cap 0.30；tier 門檻 —
- 穩膽：合併賠率 1.9–3.6 ∧ 命中 ≥50% ∧ 每腳 ≥60% ∧ EV>0
- 價值膽：2.5–5.5 ∧ 命中 ≥33% ∧ 平均 edge ≥0.04 ∧ EV≥0.05
- 高倍率：5.0–21.0 ∧ 命中 ≥12% ∧ EV≥0.08
- 火藥庫：≥5.0 ∧ 每腳 ≥62% ∧ edge≥0.08 ∧ EV≥0.10

**市場覆蓋**：模型有定價 match-winner＋to_win_1st_set／set_handicap／game_handicap／total_games／total_sets／BPTW（DERIVED_MODEL）＋aces/games props。但實際可落注嘅只有：(a) props value 邊、(b) chalk 大熱串（賠率 1.05–1.20、P>0.52、信心 ≥65、平注 1u）。Match-winner edge 單已降級做「參考，唔係建議」（`daily_report.py:1333-1344`）。

**衍生市場升級閘**（`daily_report.py:909-925`）：要 ≥20 條已結算 ∧ CLV>0 ∧ ROI≥0 先可以由 TRIAL 升做可落注。

### 已發現問題

1. ✅ **穩膽 tier 條件自相矛盾**：合併賠率 ≥1.9 同時命中 ≥50% — 兩隻 0.65 大熱夾埋命中 0.42<50%、賠率 2.4；想命中 ≥50% 就要賠率 <2 嘅腳，但合併又唔夠 1.9。所以最可靠嘅大熱組合**結構上永遠不合格**。實證：25 日報告入面得 06-17 一日真係出過組合（7 個），另外兩日 grep 中嘅「組合」行其實都係「夾唔出」嘅解釋文字。
2. ✅ **衍生市場閘門自鎖（雞先定蛋先）**：要 20 條已結算正 ROI 先可落注，但唔落注就唔會累積結算（只靠 0 注碼 scorecard 慢慢滲）→ sets/games/handicap 永遠出唔到閘。
3. ✅ **腳源被上游餓死**：組合只食「單注 filter 通過」嘅腳，而單注 filter 日常放行 0–4 隻，多數係 1.3–1.6 大熱 → 砌唔成任何 tier。
4. ✅ **同 NBA 嘅結構性分別**（點解 NBA 日日有得出）：
   - NBA 賭 soft book 球員 milestone props（一場籃球幾十隻候選腳）；網球主力係 sharp 定價嘅 match-winner 二元盤（一場一隻）
   - NBA edge = 估計機率 − **連水**隱含機率（`nba_math_engine.py:189-198`）— 莊家水位當埋係 edge；網球 edge 要贏**去水公平價**先算數 — 門檻天然高一截
   - NBA 機率 = L10 實證命中率（棒棒哋 80–90%）；網球 = 被回測否定咗嘅模型機率
   - NBA 冇質素分下限、冇信心下限、冇 edge-artifact 反殺、冇 longshot 反殺，仲有「湊唔夠就自動放寬」邏輯（hit 80→70、odds 1.20→1.15，`generate_nba_reports.py:1288-1308`）；網球全部都有而且唔放寬
   - NBA builder 目標明確係「砌夠 5 個組合」；網球係「合格先出」
5. ⚠️ 注意：NBA 出得多唔代表 NBA 啱 — NBA 嘅 ROI 到而家都未經結算驗證（settlement 基建 06-24 起咗但未餵到賽果）。

### 建議改動
- 穩膽 tier 改為「大熱串」定義（跟 chalk ≤1.20 回測：3 腳 ~breakeven、高命中），刪除自相矛盾嘅 odds/hit 組合。
