- 邊匹/邊隻 走樣最嚴重？佢嘅分析有咩做漏咗？
- 以數據表格呈現: | 預測排名 | 實際排名 | 偏差 | 原因 |

- Speed Map / 盤口邊緣預測準唔準？實際情況同預測差幾多？
- 場地判斷/傷缺 啱唔啱？有冇影響？
- 騎師戰術/教練調度 有冇出乎意料？
- 模型判斷有冇過度樂觀/悲觀？

### 角度 3 — SIP-DA01 Protocol 自我審計 ⚠️ (最關鍵)

> **呢步係審視「多角度裁決協議」本身有冇真正幫到分析。**

**3a. 有效性評估:**
- SIP-DA01 嘅辯論/審計 有冇改變最終決策？
  - 如果有 → 改變係正確嘅嗎？(即修訂版比原始版更接近實際結果？)
  - 如果冇 → 係因為原始決策已經夠準，定係辯論流於形式？
- 統計: SIP-DA01 改動咗嘅場次中，命中率係提高咗定降低咗？

**3b. 同現有邏輯嘅衝突檢測:**
- SIP-DA01 嘅審計有冇同現有嘅邏輯(如 SIP-RR / 基礎模型)衝突？
  - 例如: 基礎邏輯俾咗 A Grade，SIP-DA01 竟然建議替換 → 邊個啱？
  - 如果經常衝突 → 係基礎邏輯需要調整，定係 SIP-DA01 太保守/激進？
- 有冇同其他指標判斷重複勞動？

**3c. 改善建議:**
- 如果 SIP-DA01 有效 → 有冇需要微調 (e.g. 門檻太低/太高)？
- 如果 SIP-DA01 無效 → 應該修改、簡化、定完全移除邊個 Step？
- 現有邏輯需唔需要因為 SIP-DA01 嘅加入而調整？

### 角度 4 — 泛化性審計 (Generalizability Audit)

> **確保覆盤洞見唔會太單一，要對未來分析有用。**

- 呢場暴露出嘅問題係**普遍性**嘅（會喺其他重覆出現）定**一次性**嘅（極端意外）？
- 分類:
  - 🔵 **系統性問題** (影響所有未來): e.g. 「長期高估某類型情況」
  - 🟡 **條件性問題** (特定條件下出現): e.g. 「特定場地時預測偏誤」
  - ⚪ **孤立事件** (唔需要改 Protocol): e.g. 「意外事件」
- 只有 🔵 同 🟡 嘅洞見先值得升級為 Design Pattern / SIP 修訂

## Step 4: 結果判定

### 4a. 通過
若達到黃金標準、良好結果或最低門檻 → 進入 **Step 4c**(持續改善掃描),然後 Step 5(一致性覆核)。

### 4b. 失敗
若未達標且冇豁免:

**缺口分析 (Gap Analysis)**:
1. 邊匹預測 Top 4 嘅馬跑差咗?預測評級 vs 實際名次
2. 邊匹實際前三名嘅馬被 Analyst 忽略咗?預測評級 vs 實際名次
3. 新 SIP 規則有冇正確觸發?若有觸發但結果仍差 → 規則本身可能有問題
4. 是否 SIP 更新引入咗新嘅偏差?

**邏輯修訂建議**:
- 提出具體嘅邏輯增加或移除建議
- 指明具體嘅 resource 檔案同 Step/規則
- 提交畀用戶審批

**強制清除重做**:
- 清除失敗嘅分析結果
- 等待用戶批准邏輯修訂
- 套用修訂後重新跑 Step 2(同一場)

> ⚠️ **熔斷機制**:同一場連續失敗 3 次 → 停止並通知用戶:「Race [N] 經過 3 輪修正仍未達標。建議人工審視 SIP 更新嘅合理性。」

### 4c. 持續改善掃描 (Continuous Improvement Scan) — 通過後強制執行
> [!IMPORTANT]
> **即使場次已通過(Top 4 命中 Top 4),此步驟仍為強制性。** 唯一可跳過嘅情況為:
> - 🏆 已達黃金標準(Top 3 全入前三)— 無需進一步改善
> - 建議嘅改動會嚴重擾亂引擎其他場次嘅預測(需引用具體嘅交叉迴歸風險)

**目的:** 即使命中率已通過,引擎仍可能存在排名順序偏差、近距離遺漏、或微調空間。Validator 必須主動搵出可改善之處,而非靜默通過。

**掃描清單(逐項檢查):**

#### 4c-1. 排名順序精準度 (Ranking Order Accuracy)
- Pick 1/2 嘅實際名次是否優於 Pick 3/4?
- 若 Pick 3/4 實際名次反而優於 Pick 1/2 → 標記為「排名順序偏差」
- **輸出:** 建議引擎點樣改善排名邏輯(例如:某個維度權重應上調/下調)
- **豁免:** 若排名差距 ≤1 名(例如 Pick 2 跑第3、Pick 3 跑第2)→ 視為可容忍微差

#### 4c-2. 近距離 False Negative 掃描 (Near-Miss Detection)
- 有冇實際入前三但被 Analyst 評為 B- 或以下嘅馬匹?
- 若有 → 分析此馬嘅哪個維度被引擎低估(段速?穩定性?練馬師訊號?)
- **輸出:** 記錄為「微調候選 SIP」但**不強制修改**(需積累 ≥2 場相同模式才提出正式 SIP)

#### 4c-3. 黃金標準差距分析 (Golden Gap Analysis)
- 若僅達「良好結果」或「最低門檻」→ 分析差幾步先可以達到黃金標準
- 具體指出哪匹應入 Top 4 但被遺漏嘅馬,以及引擎邊條規則導致遺漏
- **輸出:** 「若引擎調整 [X 規則],本場有機會從 [最低門檻] 升至 [良好結果/黃金標準]」

#### 4c-4. 例外保護 (Exception Guard)
- 若掃描產出嘅改善建議會觸發已知嘅交叉迴歸風險(例如:放寬某規則會導致其他場次 False Positive 增加)→ **標記為「觀察中」而非「建議修改」**
- 極端冷門(賠率 >50x)入位導致嘅近距離 False Negative → 自動標記為「不可預見」,唔計入改善建議

#### 4c-5. 冷門命中追蹤掃描 (Longshot Hit Tracker)
> **目的:** 專門識別跑入前三但未被 Top 4 推薦嘅馬匹,分析 14.2C 冷門掃描係咪應該捕捉到佢哋。

**強制檢查:**
1. 列出所有**實際前三名但未入 Top 4 推薦**嘅馬匹
2. 逐匹檢查以下問題:
   - 該馬嘅 14.2C 冷門掃描有冇觸發?若有但被維持 D 級 → 掃描門檻可能過嚴
   - 該馬有冇觸發任何微調正面因素(14.2B)但被封頂/覆蓋規則壓制?
   - 該馬具備邊啲「冷門特徵」?(輕磅/好檔/轉倉/配備變動/密出/質新馬等)
3. **輸出:** 若識別到 14.2C 應該但未能捕捉嘅模式 → 記錄為「冷門掃描盲區 (Cold Scan Gap)」並附建議:
   - 建議將哪項因素加入 14.2C 觸發表
   - 標記為「觀察中」直到累積 ≥2 場相同冷門模式才可提出正式 SIP
4. **格式:**
```
🔍 冷門命中追蹤 — Race [N]
- 漏網冷門:#馬名 (評級 X → 實際 Yth, ZZx賠率)
  - 14.2C 觸發狀態:[已觸發但維持/未觸發]
  - 冷門特徵:[輕磅/好檔/轉倉/...]
  - 掃描盲區:[描述 14.2C 未能捕捉嘅原因]
  - 建議:[觀察中 / 建議加入 14.2C]
- 無漏網冷門 → ✅
```

**輸出格式:**
```
📈 持續改善掃描 — Race [N]
- 排名順序:[✅ 正確 / ⚠️ 偏差 — Pick X 超越 Pick Y]
- 近距離遺漏:[✅ 無 / ⚠️ #馬名 (評級→實際名次) — 低估維度: X]
- 黃金標準差距:[✅ 已達 / ⚠️ 差 X 步 — 建議: Y]
- 微調候選 SIP:[無 / SIP-XX 觀察中]
```

#### 4d. 觀察項登記簿 (Observation Register)
> [!IMPORTANT]
> 每場驗證完畢後**強制執行**。目的係集中記錄所有觀察中嘅模式,確保跨場次追蹤,避免重複登記。

**觀察項文件:** `resources/observation_log.md`(持久記錄)

**Python 自動化管理（強制）:**
```bash
# 新增觀察項
python .agents/scripts/observation_log_manager.py "resources/observation_log.md" --action add --pattern "[模式描述]" --case "[日期|場次|馬匹|結果|賠率]"

# 新增案例到現有觀察項
python .agents/scripts/observation_log_manager.py "resources/observation_log.md" --action add-case --id OBS-001 --case "[日期|場次|馬匹|結果|賠率]"

# 檢查畢業候選
python .agents/scripts/observation_log_manager.py "resources/observation_log.md" --action check-graduation
```
> 自動處理去重（≥70% 重疊）、畢業門檻（≥3 案例）、狀態轉換。
> LLM 只需提供觀察項內容，**嚴禁手動編輯 observation_log.md**。

**流程:**
1. **讀取現有觀察項**:分析前先讀取 `resources/observation_log.md`,了解已登記嘅觀察項
2. **比對新發現**:本場 Step 4c 掃描產出嘅任何新模式,逐項與現有觀察項比對:
   - **已存在** → 將本場案例加入該觀察項嘅「案例列表」(累積證據)
   - **新模式** → 新增觀察項條目
3. **畢業檢查**:若觀察項累積 **≥3 個獨立案例** 且命中率 ≥30% → 標記為「🟢 可提出正式 SIP」
4. **更新文件**:將變更寫入 `resources/observation_log.md`

**觀察項文件格式:**
```markdown
# 觀察項登記簿 (Observation Log)
> 最後更新: [日期] | 累計觀察項: [N]

## 🟡 觀察中

### OBS-001: [觀察項名稱]
- **模式描述:** [描述觀察到嘅模式]
- **潛在 SIP 方向:** [若正式化會如何修改引擎]
- **案例:**
  | # | 日期 | 場次 | 馬匹 | 評級→實際 | 賠率 |
  |:--|:---|:---|:---|:---|:---|
  | 1 | ... | R[N] | #X 名稱 | D→2nd | 22x |
- **累計案例:** [N] | **畢業門檻:** ≥3 案例 + ≥30% 命中
- **狀態:** 🟡 觀察中 / 🟢 可提出SIP / ✅ 已正式化 / ❌ 已否決

## ✅ 已正式化
[已轉為正式SIP嘅觀察項移至此處]

## ❌ 已否決
[經觀察後確認無效嘅觀察項移至此處]
```

**去重邏輯:**
- 比對維度:觀察項嘅「模式描述」核心關鍵詞
- 若新發現與現有觀察項嘅模式描述 ≥70% 重疊 → 視為同一觀察項,加入新案例
- 若新發現涉及完全唔同嘅維度/觸發條件 → 視為新觀察項

## Step 5: 一致性覆核 (Consistency Check)
通過門檻後,**必須**再跑一次分析(第 2 次)以驗證結果穩定:

1. 重新呼叫 Analyst 分析 Race [N](唔改任何規則)
2. 比較兩次分析嘅 Top 4:
   - **穩定**:Top 4 馬匹相同(順序可以唔同)→ ✅ 確認通過
   - **大致穩定**:Top 4 中 3 匹相同 → ⚠️ 通過但記錄偏差
   - **不穩定**:Top 4 中 <3 匹相同 → ❌ 記錄為「不穩定」,通知用戶

## Step 6: 進入下一場(需用戶確認)
Race [N] 通過一致性覆核後,**必須**向用戶顯示驗證結果**連同改善建議**並等待確認:

```
✅ Race [N] 盲測驗證通過。
- 🏆 黃金標準: [✅/❌]
- ✅ 良好結果: [✅/❌]
- ⚠️ 最低門檻: [✅/❌]
- 一致性覆核: [✅/⚠️]
- 📈 持續改善掃描: [結果摘要]

🔍 改善建議 (Post-Reanalysis Advisory):
[建議內容 — 見下方邏輯]

是否繼續進行 Race [N+1] 嘅盲測驗證?(Y/N)
```

### 6a. 改善建議邏輯 (Post-Reanalysis Improvement Advisory)
> [!IMPORTANT]
> **每場完成 re-analysis 後強制執行。** 目的係畀用戶清晰嘅指引:應該繼續改善本場分析,定係放手前進下一場。

根據本場驗證結果,輸出以下三類建議之一:

**🟢 建議改善 (Recommend Further Improvement):**
觸發條件(滿足任一):
- 達最低門檻但未達黃金標準,且 Step 4c 持續改善掃描識別到可行嘅微調
- 存在排名順序偏差(Pick 3/4 超越 Pick 1/2)且有明確嘅風險因素可解釋
- 存在近距離 False Negative(B- 或以下嘅馬入位),且偏差模式與已知 SIP 可糾正

輸出格式:
```
🟢 建議繼續改善本場分析:
- 改善方向:[具體描述,例如「#X 馬嘅 [維度] 被低估,若調整 [規則] 可從最低門檻升至良好結果」]
- 預期效果:[最低門檻→良好結果 / 良好結果→黃金標準]
- 風險評估:[此調整對其他場次嘅影響]
```

**🟡 放手前進 (Let Go and Move On):**
觸發條件(滿足任一):
- 已達黃金標準 — 無需改善
- 場次涉及極端冷門(冠軍/上名馬賠率 >50x)— 數據噪音,不值得過度擬合
- Step 4c 掃描產出嘅改善建議會觸發交叉迴歸風險 — 改善此場會損害其他場次
- 同一場已嘗試 ≥2 次改善但效果有限 — 邊際效益遞減

輸出格式:
```
🟡 建議放手前進:
- 原因:[極端冷門/交叉迴歸風險/邊際效益遞減/已達黃金標準]
- 此場偏差定性:[數據噪音 / 極端個案 / 引擎盲區(已有SIP覆蓋)]
```

**🔴 強制停止 (Must Stop):**
觸發條件:
- 進一步改善會需要修改引擎核心邏輯(如評級矩陣門檻、核心維度定義)— 超出微調範圍
- 改善方向與 ≥2 個現行 SIP 衝突

輸出格式:
```
🔴 建議停止改善:
- 原因:[需修改核心邏輯 / 與多個SIP衝突]
- 影響範圍:[列出可能受影響嘅 SIP]
```

- 若用戶確認 → 進入 Race [N+1],重複 Step 2-5
- 若用戶拒絕 → 直接進入 Step 7 生成總結報告
