# 數據搜尋與防錯協議 (Data Search & Verification Protocol)

本文件定義所有 NBA 數據提取嘅搜尋規則、驗證機制及來源優先級。Extractor 必須嚴格遵守。

---

## 1. 搜尋來源優先級

| 優先級 | 來源 | 用途 | 備註 |
|--------|------|------|------|
| 🥇 | ESPN | L10 數據、傷病報告、新聞 | 最可靠嘅綜合來源 |
| 🥈 | Basketball Reference | 歷史數據、H2H、DvP | 最詳細嘅統計數據庫 |
| 🥉 | CBS Sports / Yahoo Sports | 先發陣容、傷病名單 | 交叉比對用 |
| 4 | NBA.com | 官方傷病報告、賽程 | 官方權威 |
| 5 | Rotowire / FantasyPros | 預計先發、上場時間預測 | 即時陣容更新最快 |

**嚴禁使用**：個人 Blog、未經驗證嘅社交媒體帖子、賭博網站嘅「分析」。
**不要使用** `site:` 過濾器進行搜尋。

---

## 2. 核心搜尋規則

### 2-A: 直接且明確嘅搜尋指令
- 搜尋 L10 數據：`"[Player Name] stats last 10 games [PTS/REB/AST] [Month Year]"`
- 搜尋傷病：`"[Team Name] injury report [Date]"`
- 搜尋陣容：`"[Team A] vs [Team B] starting lineup [Date]"`
- 搜尋新聞：`"[Player Name] news latest [Date]"` 或 `"[Team Name] news today [Date]"`
- 搜尋 DvP：`"defense vs position NBA [YYYY] [position]"`

### 2-B: 多來源交叉比對 (Multi-Source Verification)
- **最低要求**：每個關鍵數據點至少 2 個獨立來源確認。
- 若兩個來源嘅數據有明顯出入（差異 > 15%），必須搜尋第 3 個來源作仲裁。
- L10 逐場數據特別容易出錯，必須逐場核對。

### 2-C: 日期對齊驗證 (Date Cross-Check)
1. 獲取球員 L10 數據後，**強制將最新一場比賽日期**與今日日期對比。
2. 若最新比賽日期距離今日超過 **10 天**，表示數據可能過舊或抓錯球員。
3. 必須放棄該結果並重新搜尋（加上當前月份或更具體嘅關鍵字）。

### 2-D: 合理性防呆 (Sanity Check)
- 若 L10 均值與賽季場均偏差超過 **±40%**，極有可能抓錯欄位或球員。
- 例如：賽季場均 25 分，但 L10 均值只有 12 分 → 🚨 數據可疑
- 必須作廢並重新搜尋。

---

## 3. 搜尋失敗處理

| 失敗情境 | 處理方式 |
|----------|----------|
| 連續搜尋 3 次仍無結果 | 停止搜尋，標記 `N/A (數據不足)` |
| 數據明顯過舊 (>10 天) | 換關鍵字重試 1 次，仍失敗則標記 `N/A` |
| 多來源數據衝突 | 以最權威來源 (ESPN/BBRef) 為準 |
| 球員近期被交易 | 標記新球隊，舊隊數據仍可使用但需註明 |

---

## 4. 圖片輸入限制
- **嚴禁使用用戶截圖作為統計數據來源**。
- 截圖僅用於確認「用戶想分析哪場比賽與哪些盤口」。
- **唯一例外**：可從截圖讀取讓分盤 (Spread) 數字。
- 所有其他數據必須透過網路搜尋即時獲取。

## 5. 新聞搜尋協議 (News Search Protocol)

### 搜尋範圍
- 時間窗口：過去 **72 小時**內嘅新聞
- 覆蓋範圍：每場賽事雙方球隊 + 每位候選球員

### 球隊新聞搜尋
- `"[Team Name] news today [Date]"`
- 重點關注：教練變動、戰術調整、球隊內部不和、連勝/連敗趨勢、季後賽/擺爛動態

### 球員新聞搜尋
- `"[Key Player] news latest [Date]"`
- 重點關注：傷病更新、狀態趨勢（火熱/低迷）、角色變更、場外事件、罰款/禁賽

### 輸出格式
將所有新聞整理為 `NEWS_DIGEST`，每條新聞包含：
- 新聞摘要（1-2 句）
- 來源網站
- 日期
- 影響評估標籤：`[正面]` / `[負面]` / `[中性]`（純事實標記，不做分析）

## 6. 多場賽事數據隔離規則 (Multi-Game Data Isolation Rules) [最高優先級]

> [!CAUTION]
> **此規則針對同日多場賽事嘅數據提取。LLM 喺處理第 2 場及以後賽事時，極度容易出現「數據串場污染 (Cross-Game Contamination)」同「搜尋疲勞幻覺 (Search Fatigue Hallucination)」。以下規則為強制性防護。**

### 6-A: 已知幻覺模式 (Known Hallucination Patterns)

| 模式 | 描述 | 典型例子 |
|------|------|----------|
| **數據串場 (Cross-Game Bleed)** | 將 Game 1 球員嘅數據錯誤套用到 Game 2 球員身上 | Game 1 搜過 Curry 嘅 3PT 數據，Game 2 錯誤將類似數據套入 Haliburton |
| **記憶替代搜尋 (Memory Override)** | 因為 context window 已充斥大量數據，LLM「覺得記得」而跳過 Web Search | 冇搜就直接填入 LeBron 場均 27 分（實際本季可能已變） |
| **陣容時空錯亂 (Roster Time Warp)** | 用過時嘅陣容資訊（如交易前嘅球隊） | 將已被交易嘅球員列入舊球隊先發 |
| **統計張冠李戴 (Stat Misattribution)** | 將一位球員嘅統計誤植到另一位球員 | Jokic 嘅助攻數被錯誤降低至角色球員水平 |
| **L10 複製粘貼 (L10 Copy-Paste)** | 重複使用同一組 L10 數據（可能係前一場嘅某位球員） | 兩位不同球員出現完全相同嘅 L10 數列 |

### 6-B: 強制防護措施

1. **每場新賽事 = 全新搜尋循環**：嚴禁複用前一場嘅任何搜尋結果。即使係同一位球員（如兩場都有 LeBron），都必須執行獨立搜尋。
2. **搜尋關鍵字強制包含賽事標識**：所有搜尋必須包含對陣雙方（如 `"LeBron James stats Lakers vs Celtics"`），防止抓到錯誤場次嘅數據。
3. **L10 數據唯一性檢查**：同場賽事內，若兩位球員嘅 L10 數列完全相同 → 🚨 **100% 係幻覺**，必須作廢重搜。
4. **賽事邊界標記**：每場賽事數據提取前後必須加入清晰邊界：
   ```
   ═══════════════════════════════════════
   📋 GAME X: [Team A] vs [Team B] — 數據提取開始
   ═══════════════════════════════════════
   ```

### 6-C: 額外強制規則（從 SKILL.md 內嵌升級）

5. **上下文重置聲明 (Context Reset Declaration)**：開始每場新賽事前，必須在內部明確聲明：`⚠️ 進入 Game X 數據提取 — 前序賽事數據不可引用`。
6. **嚴禁記憶替代搜尋 (No Memory Substitution)**：處理第 2 場及以後嘅賽事時，LLM 嘅 context window 已經充斥住前幾場嘅球員數據。**你嘅內部記憶極度不可靠**，必須對每位球員執行獨立 Web Search，嚴禁因「覺得記得」而跳過搜尋。即使你認為你已經「知道」Jokic 嘅助攻數或 LeBron 嘅得分數，**你都必須搜尋確認**。
7. **球員-球隊映射鎖定 (Player-Team Mapping Lock)**：提取每位球員數據前，必須先確認該球員嘅**當前球隊 (2025-26 賽季)**。LLM 內部記憶經常將球員鎖定喺舊球隊，必須以搜尋結果為準。
8. **批次完成自檢 (Per-Game Completion Check)**：每場賽事數據提取完成後，必須逐位球員快速回顧：
   1. 該球員確實屬於本場賽事嘅其中一隊？
   2. L10 數據嘅最新場次日期合理？（非前一場賽事嘅球員數據）
   3. 數據數值在合理範圍內？（例如 Jokic AST 唔會突然變成 2.0）

---

## 7. QA 雙重驗證印記

每位球員嘅數據卡完成後，必須在 `QA 雙重驗證印記` 欄位填寫以下其中一項：
- ✅ `多個來源數據完全吻合，且無指針錯位` — 數據可信
- ⚠️ `僅單一來源驗證，數據可能有偏差` — 需謹慎
- ❌ `多來源數據衝突，已採用最權威來源` — 標明衝突細節
