# 第一部：遊戲概要

## 1.1 遊戲總覽

| 項目 | 設定 |
|------|------|
| 名稱 | 旺財街機 (Wong Choi Arcade) |
| 類型 | 像素風 2D 橫向捲軸賽馬 × 情報分析策略遊戲 |
| 入口 | Dashboard 第三個 Nav Pill「🕹️ 街機」|
| 模式 | 🎮 單機 (1人) + 👥 本地多人 (2-4人) + 🌐 線上多人 (未來) |

**定位**：不只是賽馬動畫 — 而是一個結合 Wong Choi 分析方法論嘅「模擬投注策略遊戲」。玩家需要閱讀賽前新聞、分析馬匹數據、做出基於情報嘅投注決定。

## 1.3 遊戲模式

### 🎮 單機模式
- 1 人挑戰最高紀錄
- localStorage 記錄歷史最高得分 + 最佳命中賠率

### 👥 本地多人模式 (2-4人) — Draft Pick 制
1. 設定人數 + 輸入暱稱
2. 每人 $1,000 啟動資金
3. **Draft Pick 輪流制**：
   - R1: A → B → C → D
   - R2: D → C → B → A (反轉)
   - 如此類推，類似 NBA Draft Pick
4. 下注公開可見 — 已落注嘅選擇攤開，後手可以睇到前手選擇
5. 衝線後同步顯示所有人結果
6. 10 場完畢 → 總排名 + 冠軍

### 🌐 線上多人模式 (未來規劃)
- WebSocket (Socket.IO) 房間制
- QR Code 掃碼加入
- 手機同步投注 + 電腦顯示比賽畫面
- 實時同步所有玩家嘅投注同比賽狀態

## 2.4 馬匹與騎師配對邏輯 (Horse-Jockey Assignment Logic) ⭐ 新增

每次生成賽事名單 (Race Generation) 時，會按照以下順序將騎師分配給馬匹：

1. **傳奇及超級熱門專屬配對 (Set Jockey)**：
   - 頂級賽駒擁有預設嘅**專屬騎師組合**，以100%還原現實經典拍檔。
   - **經典組合範例**：
     - 金槍六十 ↔ 何澤堯 (C.Y. Ho)
     - 浪漫勇士 ↔ 麥道朗 (J. McDonald)
     - 嘉應高昇 ↔ 潘頓 (Z. Purton)
     - 精英大師 ↔ 高雅志 (F. Coetzee)*
     - 步步友 ↔ 莫雷拉 (J. Moreira)*
     - 靚蝦王 ↔ 霍達 (R. Fradd)*
   - *(註：退役傳奇騎師已加入資料庫，但遊戲內僅限策騎其專屬嘅經典名駒)*
   - **撞騎師衝突處理 (Clash Resolution)**：如果同一場賽事有兩匹馬嘅專屬跑法皆為同一騎師（例如兩匹馬都預設由潘頓主轡），系統會將該騎師優先分配畀**系統評分 (Rating) 較高**嗰匹馬。落選嘅高班馬將由系統從剩餘的冠軍級或大師級騎師池中選出最強嘅一位作為替補。

2. **實力組 (B 級) 及以下班次配對邏輯**：
   - **能力優先分配**：剩餘未配對嘅馬匹會按能力評分由高至低排序，高分馬優先從騎師池揀選能力值較高嘅騎師（模擬現實「好馬配好騎師」）。
   - **練馬師與華將偏好 (Trainer & Local Preference)**：本地華人練馬師（例如呂健威、沈集成等）會有較高機率（~30%）優先選用本地華將（何澤堯、梁家俊等）或旗下見習騎師（如鍾易禮、黃智弘、袁幸堯、黃寶妮），以帶出減磅優勢及馬房特色。

# 第四部：投注・經濟・評級

## 4.1 投注方式 (5 種)

| 投注方式 | 挑馬 | 派彩計算 |
|---------|------|---------|
| 🏆 獨贏 (WIN) | 1 匹 | 注碼 × 賠率 |
| 🏅 位置 (PLACE) | 1 匹 | 注碼 × (WIN賠率÷3, min @1.1) |
| 🔄 連贏 (Q) | 2 匹 (跑 1,2名) | 注碼 × (A賠率 × B賠率 × 0.5) |
| 🥉 位置Q (WP) | 2 匹 (跑入 1-3名) | 注碼 × (A × B × 0.15) ⭐ 新增極度地道玩法 |
| 🎰 三重彩-順序 (TIERCE) | 3匹 (要中順序) | 注碼 × (A × B × C × 0.3) |
| 🎰 三重彩-唔順序 (TRIO) | 3匹 | 注碼×6 × (A × B × C × 0.05) |

> 最低注碼：$10 · 最高注碼：無上限 (可 all-in 🔥)
> **派彩顯示細節**：連贏 (Q) 及三重彩 (TIERCE) 結算時不可只顯示倍數。為咗迎合香港馬迷體驗，UI 必須將賠率轉換為 **「$10 一注派幾多」** 嘅經典格式（例如：連贏 3, 8 派彩 $315.50）。

## 4.2 賠率生成算法

```
綜合能力分 = baseSpeed×25 + stamina×15 + finalSprint×15
           + consistency×10 + energyLevel×10
           + 騎師能力×15 + 練馬師能力×10
勝率 = 該馬能力分 / 全場12匹總和
原始賠率 = 1 / 勝率
最終賠率 = 原始賠率 × 0.82 (馬會利潤)
限制：@1.5 ≤ 賠率 ≤ @99.0
```

泥地賽修正：baseSpeed 乘以 `dirtPreference` 再代入公式。

## 4.3 破產及借貸機制 (大耳窿救亡)

| 項目 | 設定 |

| 玩家選擇 | 接受 → 獲 $500 繼續玩；拒絕 → 直接 Game Over 提早進入總結 (F 級)。 |
| 評級懲罰 | 每次借貸 = 最終評級分數強制扣 500 分 |

## 16.2 物理引擎速度公式

```javascript
// Phase 1: 出閘 (0-10%)
speed = gateSpeed × baseSpeed × random(0.9, 1.1)

// Phase 2: 中段 (10-65%)
speed = baseSpeed × consistency × random(0.95, 1.05)
stamina -= (1 - energyLevel/100) × 0.5  // 每 tick
if (random() < burstChance) speed *= burstPower  // 持續 1-3 秒

// Phase 3: 彎位 (65-75%)
if (barrierDraw <= 4) speed *= 1.05   // 內欄加速
if (barrierDraw >= 9) speed *= 0.95  // 外檔懲罰

// Phase 4: 直路 (75-100%)
speed += finalSprint × (stamina / 100)
if (stamina < 20) speed *= 0.7        // 攰馬懲罰
// 騎師技能喺呢度觸發
```

### 三/四疊偵測 (每 tick)
```javascript
for (each horse) {
  sameRow = horses.filter(h => Math.abs(h.y - horse.y) < 0.5)
  if (sameRow.length >= 3 && horse.runningStyle !== '領放') {
    horse.speed *= random(0.75, 0.85)  // 被困懲罰
    horse.trappedTimer = random(60, 180) // 1-3 秒 @60fps
  }
}
```

### 泥地修正
```javascript
if (trackSurface === 'dirt') {
  speed *= horse.dirtPreference  // 0.5-1.5
  staminaDrain *= 1.15
  if (runningStyle === '領放') speed *= 1.08
  if (runningStyle === '後上') speed *= 0.90
}
```

# 第十七部：核心系統架構與效能規範 (開發團隊必讀) ⭐ 新增

開發團隊喺落手寫 Code 之前，必須嚴格遵守以下架構準則，確保遊戲效能、擴展性及多人在線功能嘅穩定：

## 17.1 Canvas 與 React 狀態徹底解耦 (Decoupling)
- **問題**：React 嘅狀態更新 (setState) 會觸發重新渲染 (Re-render)。如果將 60FPS 嘅 Game Loop 同 React State 綁埋一齊，過度渲染會導致嚴重掉幀 (Drop frames) 及手機發熱。
- **規範**：
  - Game Engine (Canvas 物理運算、渲染) 必須獨立成純 JS Class / Hook，並利用 `useRef` 直接操作 DOM。
  - Game Loop 內絕對**禁止**呼叫高頻次嘅 React `setState`。
  - React 只負責管理 UI (例如賽前投注、結算畫面)。比賽進行中，Game Engine 僅限每 0.5-1 秒左右透過 Callback 更新一次外部 UI (例如傳送 Live Comment 事件字串)。

## 17.2 必須使用「固定種子」隨機數 (Seeded RNG)
- **問題**：原生物理公式高頻率使用 `random()` 或 `Math.random()`。喺原生隨機體系下，無法做到「同一場賽道、同一個結果」，對未來嘅「線上多人同步」及「賽事回放 (Replay)」是致命傷。
- **規範**：
  - 引入具備 Seeded RNG 功能嘅庫 (例如 `seedrandom`)。
  - 每場比賽開始前，根據 Race ID 或 Timestamp 生成一個**全局唯一嘅 Race Seed**。
  - 引擎內所有隨機判定 (例如：爆發機率、意外發生率、起步快慢) 必須統一呼叫呢個 Seeded RNG，確保物理運算 100% 決定性 (Deterministic)。

## 17.3 資產優化：棄用 Base64，全面轉向 Sprite Sheet
- **問題**：雖然 Base64 減少 HTTP 請求，但龐大嘅 170+ 匹馬數據同幾十款綵衣 Sprite 會令 JS Bundle Size 大幅膨脹，嚴重拖慢載入時間。
- **規範**：
  - 捨棄 `sprites.js` 嘅 Base64 做法，改用 **Texture Atlas (精靈圖集)** `.png` 格式。
  - 配合 Canvas `drawImage(image, sx, sy, sWidth, sHeight, dx, dy, dWidth, dHeight)` 進行實時裁剪 (Clipping) 渲染。
  - 所有素材喺 ArcadePage 初始化時統一 Preload，確保賽事開始時畫面零遲延。

## 17.4 手機板 HUD 密度降級機制 (Graceful Degradation)
- **問題**：iPhone 等窄屏設備 (375px) 難以同時清晰容納「賽道畫面、即時排名、棟哥貼士、Live 評述」四大元素。
- **規範**：
  - **手機直屏**：自動隱藏或摺疊「即時排名面板」，改為僅喺馬匹頭頂顯示名次數字。
  - Live 評述面板縮減為單行跑馬燈，或以半透明彈幕形式覆蓋喺賽道頂部 / 底部。
  - 棟哥貼士僅限於「投注階段」顯示，開閘起步後自動切換至隱藏狀態，留空視野。

## 17.5 物理引擎必須使用時間差 (Delta Time)
- **問題**：原生物理公式如果單純依賴「每 tick (幀)」更新，會受玩家設備螢幕刷新率 (Refresh Rate) 影響。144Hz 螢幕下嘅馬匹會比 60Hz 螢幕跑得快 2.4 倍，導致遊戲完全失去公平性同準確性。
- **規範**：所有涉及位移、速度計算、Stamina 扣減等狀態更新邏輯，必須乘上 `deltaTime` (上一幀到目前經過嘅毫秒數)。令物理運算結果與 Frame Rate 徹底脫鉤 (`x += speed * deltaTime`)。

## 17.6 繞過瀏覽器音效阻擋機制 (Autoplay Policy)
- **問題**：Chrome 及 Safari 預設禁止網頁喺未經使用者互動 (User Interaction) 的情況下強制播放聲音。如果載入即播 BGM，會觸發 Error 並導致成個音頻系統被封鎖。
- **規範**：進入 Arcade 主畫面首頁前，必須設計一個明確嘅「Click To Start (點擊進入街機)」按鈕。透過玩家該次點擊事件去 Initialize `AudioContext`，解鎖後續嘅起步鐘聲同衝線歡呼音效。

## 17.7 多人選馬死鎖防範 (3 分鐘強制推進 Timer)
- **問題**：喺本地多人 Draft Pick 模式（輪流選馬）之下，如果其中一位玩家猶豫太耐或者行開咗，成個遊戲進程就會被無限期卡死無得玩。
- **規範**：投注/選馬階段必須加入一個強制性嘅**「3 分鐘 (3 Minutes) 倒數 Timer」**。夠鐘如果目前順序嘅玩家仲未確認落注，系統會自動幫佢放棄選擇（或派發隨機馬匹），強制推進到下一位，嚴防遊戲死鎖。

## 17.8 規避 JavaScript 浮點數派彩誤差
- **問題**：JS 處理浮點數 (IEEE-754) 會有精度誤差（例如 `$10.5 * @3.15` 可能得出 `$33.075000000000001`），直接顯示落 UI 會非常難睇，甚至引發內部 Balance 對數錯亂。
- **規範**：所有牽涉到注碼、賠率轉換及派彩結算 (Balance) 嘅變化，完成計算後一律必須透過 `Math.round()` 或 `Math.floor()` 強制抹除小數點尾數，保持遊戲貨幣系統整數化。
